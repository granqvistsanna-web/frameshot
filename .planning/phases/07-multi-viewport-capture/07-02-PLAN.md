<objective>
Push the multi-viewport loop INTO `runCapture` per D-03 so both the CLI and the server stay dumb adapters. Refactor `launchBrowser` to accept a per-viewport entry so it can be invoked N times from inside the loop, each call producing its own browser + context. Every progress event gains `viewport: <name>` scoping. Return shape changes from a single object to an array.

Purpose: The presentation-free seam (commit 220ff9f) was extracted precisely so the lifecycle + progress-stream logic lives in one place. Per CONTEXT.md D-03, "pushing the loop into runCapture keeps that contract clean: callers stay dumb adapters." The alternative — CLI/server loops calling `runCapture` per viewport — duplicates lifecycle and SSE/spinner wiring across two callers, exactly what extracting the seam was meant to prevent. ROADMAP success criterion #3 ("Each viewport runs in its own Playwright browser context — no shared cookies/storage/animation-shim state") implies a fresh `chromium.launch` per viewport (not just `newContext`), so the loop calls `launchBrowser` N times.

Output: `runCapture` returns `Array<{ outputPath, hideSummary, viewportName }>`. `launchBrowser` takes a second parameter. The prepare pipeline applies per-context (per-viewport) unchanged. The CLI + server wiring updates are deferred to Wave 3 — this plan does NOT touch them.
</objective>
