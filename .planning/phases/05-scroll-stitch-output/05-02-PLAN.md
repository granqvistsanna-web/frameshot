<objective>
Build the sharp composite module — `src/capture/stitch.js`. This is the OUT-02 deliverable: take the ordered `Buffer[]` of viewport-sized PNGs and the `geometry` object produced by `captureFrames` (05-01), and assemble them into ONE full-page PNG Buffer using sharp's `composite()` API. The function is a pure transform — `Buffer[] + geometry → Buffer`, no I/O, no browser handles, no fs.

The key correctness rules are encoded by the math:

1. **Canvas dimensions are physical pixels.** Each input frame is `viewportWidth · DSR × viewportHeight · DSR` (because `page.screenshot` defaults to `scale: 'device'` and 05-01 keeps that explicit). The canvas must match: `canvasWidth = Math.round(viewportWidth · deviceScaleFactor)`, `canvasHeight = Math.round(totalHeight · deviceScaleFactor)`. `Math.round` defends against fractional DSR (1.25, 1.5) producing non-integer pixel coordinates that sharp would reject (Pitfall 8).
2. **Overlay offsets multiply by DSR.** `frameYOffsets[i]` is CSS pixels (because that's the unit `scrollTo` and `clip` use in 05-01). The overlay's `top` must be physical pixels: `top: Math.round(frameYOffsets[i] · deviceScaleFactor)`, `left: 0`. The mapping is uniform across all frames.
3. **Order is the overlap correctness mechanism.** Sharp's `composite()` applies overlays in array order; later overlays draw ON TOP (node_modules/sharp/lib/index.d.ts:312-318 — documented surface contract, NOT a quirk). When 05-01's last frame is clamped to `totalHeight - viewportHeight` (the overlap-and-trust-composite strategy), that frame appears LAST in `frames[]` and `frameYOffsets[]`, so sharp paints it AFTER the prior frame — the overlap region is cleanly overwritten with the last frame's correct pixels. This module just respects the array order; the strategy lives at the contract between 05-01 (clamp) and the documented sharp behavior.
4. **Pass frames directly — no pre-wrap.** `overlays = frames.map((input, i) => ({ input, top: Math.round(frameYOffsets[i] · DSR), left: 0 }))`. `input` is the PNG Buffer itself. Wrapping each with `sharp(buf)` (Risk 20) is wasted decode+re-encode work — sharp's composite accepts Buffer directly per the OverlayOptions type (index.d.ts:1487-1510).
5. **Blank canvas is transparent RGBA.** `sharp({ create: { width: canvasWidth, height: canvasHeight, channels: 4, background: { r: 0, g: 0, b: 0, alpha: 0 } } })`. Every pixel will be overwritten by the first composite anyway, but transparency is the safest default — any rounding-induced sliver (the overlap strategy precludes them, but belt-and-braces) shows transparent rather than uninitialized memory.
6. **Encoder defaults.** `.png().toBuffer()` — NO custom `compressionLevel`, `progressive`, `palette`, or `sharp.cache(false)`/`sharp.concurrency(N)` calls (Risk 15). v0.1 personal tool; tuning is premature.

Purpose: OUT-02 ("Captured frames are stitched into a single PNG using sharp with no ghosted sticky elements and no visible seams") and SC #3 from ROADMAP Phase 5 ("Sticky navigation elements appear exactly once at the top of the stitched image, not repeated at every viewport-height interval") both ride on the stitcher. SC #3 is actually a Phase 4 success — if the sticky nav is in `prepare.hide`, Phase 4 removes it before 05-01 captures, so it physically does not appear in any frame. The stitcher itself does NOT scan for sticky elements (Risk 18 / Pitfall 2). The "no visible seams" property comes from the overlap-and-trust-composite contract: there ARE no seams because the last frame OVERLAPS the prior frame (when totalHeight is not an exact multiple of viewportHeight) and sharp's composite-order overwrites the overlap with correct content. The math makes seams structurally impossible — no per-pixel comparison, no row-blending, no visual heuristics needed.

This plan is wave 1 (parallel-authorable with 05-01) because the two files' `files_modified` sets don't overlap (`src/capture/stitch.js` vs `src/capture/frames.js`) and the contract between them — the `{ frames: Buffer[], geometry: { ..., frameYOffsets, deviceScaleFactor } }` shape — is locked by 05-01's must_haves and consumed here. Runtime end-to-end verification of the chain is owned by 05-03 (wave 2). This plan's `<verify>` is self-contained: synthesize a small set of PNG frame buffers via sharp itself, run them through `stitchFrames`, decode the output, and assert the canvas dimensions + the per-pixel color at specific known positions (verifies overlay positioning AND overlap-overwrite correctness).

Output: `src/capture/stitch.js` — one exported async function `stitchFrames(frames, geometry)`, ONE import (`import sharp from 'sharp'`).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `frames: Buffer[]` from 05-01 → sharp composite input | Each Buffer is a PNG produced by Playwright's `page.screenshot`. sharp decodes the PNG natively via libvips. No user input flows into the buffer beyond the screenshot's pixel content. |
| `geometry: { ... }` from 05-01 → canvas dimensions + overlay offsets | All five fields are numbers read from `window` properties in 05-01's `page.evaluate`. The Page is sandboxed; the values are bounded by the browser's actual viewport and document. |
| sharp `create` config → libvips canvas allocation | `Math.round(viewportWidth * DSR)` and `Math.round(totalHeight * DSR)` are integer pixel counts. libvips allocates `width * height * 4` bytes (RGBA). | sharp `.composite()` overlay buffers → libvips decode | Each overlay Buffer is a PNG; libvips decodes natively. No untrusted attacker-controlled binary crosses this boundary (the screenshots come from the user's own browser rendering the user's chosen URL). |
| Thrown sharp errors (PNG decode failure, malloc failure, libvips internal) → caller's catch | No try/catch in this module; failures bubble to the CLI's finally block. Phase 6 (CLI-03) will format them. |
