<objective>
Build the scroll-capture loop module — `src/capture/frames.js`. This is the OUT-01 deliverable: scroll the prepared page in viewport-height steps from top to bottom, take ONE viewport-sized PNG screenshot per step, and return the ordered Buffer[] alongside the geometry the stitcher (05-02) needs to compose them into a single full-page PNG. The function is pure browser-layer orchestration — `page.evaluate` + `page.screenshot` only; NO sharp, NO fs.

The key correctness rules are encoded by the math:

1. **Geometry is read ONCE.** A single `page.evaluate` at the start returns `{ viewportWidth, viewportHeight, totalHeight, deviceScaleFactor }`. The loop body never re-reads (RESEARCH §Pitfall 5 + 05-PATTERNS Risk 6 — re-reading mid-loop introduces inconsistent canvas math and infinite-loop risk if lazy content extends the page during capture).
2. **Frame y-offsets are pre-computed** into an array `frameYOffsets`. Two cases: (a) `totalHeight <= viewportHeight` → single-frame fast path `[0]`; (b) multi-frame → push `i * viewportHeight` for `i = 0..nFull-1`, then IF `totalHeight % viewportHeight > 0` push `(totalHeight - viewportHeight)` as the clamped last frame. The clamp produces a frame that OVERLAPS the previous one — sharp composite-order in 05-02 paints the last frame on top, overwriting the overlap region cleanly (Pattern 1 lines 327-332 + Risk 10).
3. **The loop body is three operations per frame:** (a) `page.evaluate((y) => window.scrollTo({ top: y, behavior: 'instant' }), y)` — instant ONLY, never smooth (Risk 7, Pitfall 6); (b) `await page.evaluate(() => new Promise((r) => requestAnimationFrame(() => r())))` — single rAF roundtrip for paint to settle, NOT `page.waitForTimeout(200)` (Risk 8, Pitfall 3 — Phase 4's scrollPrime already triggered lazy-loads; per-frame fixed delay accumulates ~200ms × frames of waste); (c) `page.screenshot({ clip: { x: 0, y, width: viewportWidth, height: viewportHeight }, animations: 'disabled', scale: 'device', type: 'png' })` — `clip` in CSS pixels per Playwright types.d.ts:24276; `scale: 'device'` explicit because the default is 'device' BUT this is the retina-quality contract (CAP-02 — Risk 17 forbids 'css'); `animations: 'disabled'` belt-and-braces with Phase 4's CSS guards.
4. **deviceScaleFactor comes from `window.devicePixelRatio`** at capture time — NOT hard-coded to 2 (Risk 9; CAP-02 supports any DSR, schema accepts any non-zero number). The stitcher in 05-02 multiplies CSS-pixel offsets by DSR for the canvas math.

Purpose: OUT-01 ("Full-page screenshot is captured by scrolling in viewport-height steps and capturing each frame") and SC #2 from ROADMAP Phase 5 ("The output PNG's height equals the full scrollable page height and shows content from top to bottom with no missing sections") both rest on this module. The choice that distinguishes Phase 5 from Playwright's `fullPage: true` is encoded HERE: we capture clean per-viewport frames so the stitcher can compose them WITHOUT the rerender-per-pass behavior that ghosts sticky elements. Phase 4's `hide` list already removed the sticky nav before this module runs (Risk 18 — sticky-element handling is OWNED BY Phase 4's hide, not Phase 5). If the user forgets to add `nav.sticky` to `prepare.hide`, the result ghosts; that's a config bug, not a Phase 5 bug.

This plan is wave 1 (parallel-authorable with 05-02) because the two files' `files_modified` sets don't overlap (`src/capture/frames.js` vs `src/capture/stitch.js`) and the contract between them — the `{ frames: Buffer[], geometry: { viewportWidth, viewportHeight, totalHeight, frameYOffsets, deviceScaleFactor } }` return shape — is locked here and consumed by 05-02. Runtime end-to-end verification of the chain is owned by 05-03 (wave 2 — the orchestrator + CLI wiring). This plan's `<verify>` includes a self-contained runtime check against an http server fixture (no dependency on 05-02 or 05-03 landing) plus the static-source gates.

Output: `src/capture/frames.js` — one exported async function `captureFrames(page)`, ZERO imports (the Page is a parameter; the JSDoc references `import('playwright-chromium').Page` as a type only).
