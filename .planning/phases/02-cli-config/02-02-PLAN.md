<objective>
Build the validator layer that turns a YAML file path into a fully-typed config object — or a clean, field-named error message. This plan ships two pure modules (`src/config/schema.js`, `src/config/load.js`) plus two malformed fixtures used to prove CFG-02. NO CLI wiring happens here — 02-01 (wave 2) imports `loadConfig` and wires it into the commander action handler. Decoupling validation from the CLI lets this whole plan run in wave 1 alongside 02-03 (template resolver) with zero file overlap.

Purpose: CFG-02 is the most opinionated requirement in Phase 2 — invalid configs MUST be rejected with field-named messages, NEVER a raw zod dump. Isolating validation + error formatting in their own module (per RESEARCH.md §Architectural Responsibility Map) makes that rule easy to enforce and trivially testable. CFG-01 falls out of having a real schema. ROADMAP success criteria #1 and #2 become observable once 02-01 (wave 2) wires this loader into the capture action handler.

Output: `src/config/schema.js` (zod schema + `formatZodError`), `src/config/load.js` (file → parsed → validated, throws `ConfigError`), two bad-config fixtures in `samples/`.
</objective>

      ```
      export class ConfigError extends Error {
        constructor(message) { super(message); this.name = 'ConfigError'; }
      }
      ```
      Why a named subclass: lets the top-level catch in `index.js` (wired in 02-01) distinguish "user input was bad" from "we have a bug." For v0.1 the catch handler treats them the same (print message, exit 1), but the type stays useful.

    **`loadConfig(configPath)` — match RESEARCH.md §Pattern 2 four-step pipeline:**

    Step 1 — Resolve path:
      `const absPath = resolve(process.cwd(), configPath)`. Use `process.cwd()`, NOT `import.meta.url` (CLI-01 demands the user's CWD).

    Step 2 — Read file:
      ```
      let raw;
      try {
        raw = await readFile(absPath, 'utf8');
      } catch (err) {
        if (err.code === 'ENOENT') throw new ConfigError(`Config file not found: ${absPath}`);
        if (err.code === 'EACCES') throw new ConfigError(`Permission denied reading: ${absPath}`);
        throw err;  // genuinely unexpected — let it bubble
      }
      if (raw.charCodeAt(0) === 0xFEFF) raw = raw.slice(1);  // strip UTF-8 BOM (RESEARCH.md §Pitfall 3)
      ```

    Step 3 — Parse YAML:
      ```
      let parsed;
      try {
        parsed = yaml.load(raw);
      } catch (err) {
        if (err.name === 'YAMLException') {
          const line = (err.mark?.line ?? 0) + 1;   // js-yaml uses 0-indexed; display 1-indexed
          const col = (err.mark?.column ?? 0) + 1;
          throw new ConfigError(`YAML parse error at ${absPath}:${line}:${col} — ${err.reason}`);
        }
        throw err;
      }
      ```

    Step 4 — Validate:
      ```
      const result = configSchema.safeParse(parsed);
      if (!result.success) {
        const lines = formatZodError(result.error);
        throw new ConfigError(`Invalid config in ${absPath}:\n  ${lines.join('\n  ')}`);
      }
      return result.data;
      ```
      - Use `safeParse`, NOT `parse` — that's the whole point of CFG-02 (RESEARCH.md §Pitfall 5).
      - Two-space indent on continuation lines so the user can scan multiple issues at a glance.

      - Do NOT call `console.log` / `console.error` in this module — throw `ConfigError` and let the caller decide presentation.
      - Do NOT call `process.exit` — that's `index.js`'s top-level catch's job (wired in 02-01).
      - Do NOT import chalk — same reason; presentation belongs to the caller.
      - Do NOT add a logger abstraction. PROJECT.md anti-bloat principle (RESEARCH.md §What NOT To Do #2).
      - Do NOT catch `ZodError` directly anywhere — `safeParse` already gives us a structured result; the only path that could surface a raw ZodError is forgetting to use `safeParse`.
      - Do NOT call `JSON.parse(JSON.stringify(parsed))` to "normalize" — RESEARCH.md §Anti-Patterns: it strips zod's coerce and wastes cycles.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| YAML file on disk → JS object in memory | Untrusted file content crosses into application data via `yaml.load`. The schema is the gate; js-yaml v4's safe-by-default `load` (RESEARCH.md §Security Domain) is the lower-level mitigation. |
| ZodError → user-facing message | The format choice (field-named vs raw dump) is a UX trust contract: CFG-02 says we NEVER leak raw zod internals to the user. `formatZodError` is the only path; `loadConfig` only ever throws `ConfigError` with a pre-formatted `.message`. |
| ConfigError → caller's catch handler | The contract this plan offers: bad input throws `ConfigError` with a clean `.message`. The actual stderr formatting (and `process.exit`) is the caller's responsibility — 02-01 will wire the top-level catch in `index.js`. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-T | Tampering / EoP | `yaml.load` on untrusted file | mitigate | js-yaml v4's default `yaml.load` is safe-by-default — `safeLoad` was removed and the default schema no longer instantiates custom constructors that could lead to code execution (RESEARCH.md §Security Domain row 1, verified against js-yaml v4 changelog). We use exactly `yaml.load(text)` — never extend the schema. |
| T-02-02-IY | Information Disclosure | formatZodError output | mitigate | The function builds messages from `issue.path`, `issue.expected`, `issue.received`, and `issue.message` — a curated subset. It does NOT include `issue.code` (internal zod taxonomy), `error.format()` output, or stack traces. Task 2 and Task 3 verify assert the literal string `ZodError` does NOT appear in any error message. |
| T-02-02-PP | Tampering | Prototype pollution via YAML `__proto__: { admin: true }` | mitigate | js-yaml v4's default schema does not allow `__proto__` key injection during object construction (RESEARCH.md §Security Domain row 4). zod schema validation strips unknown keys anyway. |
| T-02-02-PT | Information Disclosure | Config path argument (path traversal) | accept | User is the threat boundary (PROJECT.md personal tool, RESEARCH.md §Security Domain row 3). Reading a file the user pointed at is documented behavior, not a vulnerability. |
| T-02-02-DOS | Denial of Service | Large YAML file | accept | A user pointing framershot at a multi-GB YAML would OOM Node. Accepted: personal tool, the user controls their own files. No size cap added — would be premature. |
| T-02-02-FUT-PT | Tampering | Output path template (`../../etc/passwd` in `output:` field) | accept (flagged for Phase 5) | The `output` field in the schema is `z.string().min(1)` — no traversal check.
