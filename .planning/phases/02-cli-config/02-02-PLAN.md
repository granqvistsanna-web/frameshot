<objective>
Build the validator layer that turns a YAML file path into a fully-typed config object — or a clean, field-named error message. Wire it into the existing `lib/cli.js` so `framershot capture <config>` becomes a real smoke test: valid configs print pretty JSON, invalid configs print a one-line-per-issue error and exit 1. This is the trust boundary where untrusted YAML content becomes typed application data.

Purpose: CFG-02 is the most opinionated requirement in Phase 2 — invalid configs MUST be rejected with field-named messages, NEVER a raw zod dump. Isolating validation + error formatting in their own module (per RESEARCH.md §Architectural Responsibility Map) makes that rule easy to enforce and trivially testable. CFG-01 falls out of having a real schema. ROADMAP success criteria #1 (prints resolved config) and #2 (specific error on bad config) both become observable from the shell after this plan.

Output: `lib/config-schema.js` (zod schema + `formatZodIssues`), `lib/config-loader.js` (file → parsed → validated, throws `ConfigError`), updated `lib/cli.js` (action now calls the loader and prints JSON), two bad-config fixtures.
</objective>

      export function formatZodIssues(zodError) {
        return zodError.issues.map((issue) => {

          if (issue.code === 'invalid_type') {
            return `${field}: expected ${issue.expected}, got ${issue.received ?? typeof issue.input}`;

          if (issue.code === 'too_small') {
            return `${field}: ${issue.message} (minimum: ${issue.minimum})`;
          }
          if (issue.code === 'too_big') {
            return `${field}: ${issue.message} (maximum: ${issue.maximum})`;
          }
          return `${field}: ${issue.message}`;
        });
      }

      ```
      export class ConfigError extends Error {
        constructor(message) { super(message); this.name = 'ConfigError'; }
      }
      ```
      Why a named subclass: lets the top-level catch in `index.js` (or a Phase 6 error formatter) distinguish "user input was bad" from "we have a bug." For v0.1 the catch handler treats them the same (print message, exit 1), but the type stays useful.

    **`loadAndValidateConfig(configArg)` — match RESEARCH.md §Pattern 2 four-step pipeline:**

    Step 1 — Resolve path:
      `const absPath = resolve(process.cwd(), configArg)`. Use `process.cwd()`, NOT `import.meta.url` (CLI-01 demands the user's CWD).

    Step 2 — Read file:
      ```
      let raw;
      try {
        raw = await readFile(absPath, 'utf8');
      } catch (err) {
        if (err.code === 'ENOENT') throw new ConfigError(`Config file not found: ${absPath}`);
        if (err.code === 'EACCES') throw new ConfigError(`Permission denied reading: ${absPath}`);
        throw err;  // genuinely unexpected — let it bubble
      }
      if (raw.charCodeAt(0) === 0xFEFF) raw = raw.slice(1);  // strip UTF-8 BOM (RESEARCH.md §Pitfall 3)
      ```

    Step 3 — Parse YAML:
      ```
      let parsed;
      try {
        parsed = yaml.load(raw);
      } catch (err) {
        if (err.name === 'YAMLException') {
          const line = (err.mark?.line ?? 0) + 1;   // js-yaml uses 0-indexed; display 1-indexed
          const col = (err.mark?.column ?? 0) + 1;
          throw new ConfigError(`YAML parse error at ${absPath}:${line}:${col} — ${err.reason}`);
        }
        throw err;
      }
      ```

    Step 4 — Validate:
      ```
      const result = configSchema.safeParse(parsed);
      if (!result.success) {
        const lines = formatZodIssues(result.error);
        throw new ConfigError(`Invalid config in ${absPath}:\n  ${lines.join('\n  ')}`);
      }
      return result.data;
      ```
      - Use `safeParse`, NOT `parse` — that's the whole point of CFG-02 (RESEARCH.md §Pitfall 5).
      - Two-space indent on continuation lines so the user can scan multiple issues at a glance.
