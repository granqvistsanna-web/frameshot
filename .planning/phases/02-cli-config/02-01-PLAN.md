<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| `process.argv` → commander | Untrusted strings from the user's shell cross into commander's parser. commander v14 (~140M weekly downloads, vetted in Phase 1) handles malformed argv safely. |
| config path argument → filesystem | The string after `framershot capture` is passed to `path.resolve` and (in 02-02) `fs.readFile`. PROJECT.md frames the user as the threat boundary — they can read any file they own; that's expected behavior, not a vulnerability. |
| Thrown Error → stderr | What the catch handler prints becomes user-facing output. CFG-02 (proven in 02-02) makes this a UX trust contract; this plan establishes the print-only-err.message pattern that CFG-02 depends on. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-PT | Information Disclosure | Config path argument resolution | accept | User is the threat boundary (PROJECT.md: personal tool, local-only). `path.resolve(process.cwd(), configArg)` is intentionally unrestricted — the user explicitly chose what to point at. No traversal blocking; that would break CLI-01's "from any directory" contract (RESEARCH.md §Security Domain). |
| T-02-01-IY | Information Disclosure | Top-level catch in index.js | mitigate | The catch prints `err.message` only, never the full Error object and never the stack. This is the upstream half of CFG-02's "no raw zod dump" contract — by the time 02-02 throws a ConfigError, this catch handler is already wired to print only the formatted message. |
| T-02-01-SC | Tampering | npm supply chain | mitigate | No new dependencies in Phase 2 (RESEARCH.md §Package Legitimacy Audit: zero additions). The seven packages were vetted in Phase 1's threat model; no fresh attack surface. |
| T-02-01-EE | Elevation of Privilege | Bin shebang execution | accept | Inherited from Phase 1. `framershot` runs as the invoking user; no setuid. This plan rewrites `index.js` but preserves the same execution model. |
</threat_model>
