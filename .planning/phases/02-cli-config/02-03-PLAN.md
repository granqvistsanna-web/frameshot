<objective>
Build the pure output-path template resolver. Given a string like `./screenshots/{date}/{viewport}/{page}.png` and a context `{ viewport: 'desktop', page: 'home' }`, return `./screenshots/2026-05-19/desktop/home.png`. After this plan, both 02-01 (smoke test that prints the resolved config + sample resolved output path) and Phase 5 (actual file writer) can call `resolveTemplate` without touching date or string-substitution logic themselves.

Purpose: CFG-03 requires `{date}`, `{viewport}`, and `{page}` to be supported in output path templates. Isolating this in a tiny pure function makes it trivially testable, side-effect-free (no `new Date()` leaking into callers), and ready for the smoke test in 02-01.

Output: src/output/template.js — one named export, ~20 lines of source.
</objective>

  <action>
    Create src/output/template.js as an ESM module with exactly one named export: `resolveTemplate`.

    Signature: `export function resolveTemplate(template, context)`

    Behavior:
      - `template` is a string. `context` is an object with three keys: `viewport` (string, required), `page` (string, required), and `date` (optional `Date`; if omitted or undefined, default to `new Date()`).
      - Format the date as ISO `YYYY-MM-DD` (year, month, day in local time — match the user's clock; PROJECT.md is a personal local tool, no timezone confusion concern). Use `String(date.getFullYear()).padStart(4,'0')`, `String(date.getMonth()+1).padStart(2,'0')`, `String(date.getDate()).padStart(2,'0')` and join with `-`. Do NOT use `date.toISOString().slice(0,10)` — that returns UTC, which would print yesterday's date for late-evening captures in non-UTC zones. For a daily-touch personal tool, the user's local "today" is what they expect in a folder name.
      - Replace the literal substring `{date}` with the formatted date.
      - Replace the literal substring `{viewport}` with `context.viewport`.
      - Replace the literal substring `{page}` with `context.page`.
      - Use a single regex pass (`template.replace(/\{(date|viewport|page)\}/g, ...)`) rather than three sequential `.replace()` calls. The replacement callback looks up the matched key in a small local table and returns the value. A single regex pass means the order of placeholders in the template does not matter, and a value that itself contains `{date}` (unlikely but defensive) will not be re-substituted.
      - Unknown placeholders (e.g. `{foo}`, `{viewport_name}`, `{Date}` with capital D) MUST be left untouched in the output. Do NOT silently drop them — leaving them visible makes typos surface immediately when the user looks at the resulting file path. The regex above already enforces this (it only matches the three known keys verbatim).
      - If `context.viewport` or `context.page` is undefined when the corresponding placeholder is present, throw `TypeError` with a message like `resolveTemplate: context.viewport is required when template contains {viewport}`. Be specific about which field is missing. Do NOT silently substitute the string `"undefined"`.

    Do NOT add any imports — this module has zero dependencies. Do NOT add any other exports (no default export, no helper exports). Do NOT add `node:path` joining — the resolver works on whatever string the user wrote, including relative paths starting with `./`. Phase 5 will handle absolute-path resolution and `mkdir -p` when it writes the actual file.

    Do NOT introduce a "format" or "strftime" abstraction. Three placeholders, hardcoded. The day we need a fourth (v0.2 may add `{region}` for region capture), we add one line to the small table — that day is not today.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| context object → output path string | The viewport/page strings flow from a YAML config into a filesystem path. Phase 5 will use this path with `mkdir -p` and `writeFile`. A user-controlled string with `..` or `/` in it could write outside the intended directory. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-T | Tampering | resolveTemplate output → filesystem | accept | The user authors their own config. `viewport` and `page` are values the user wrote in their own YAML — same trust level as the rest of their config. PROJECT.md frames this as a personal tool, so the threat surface is the user's own input. If Phase 5 needs to harden against path traversal, that mitigation belongs in the output writer (with `path.resolve` + prefix check), not in this pure string substitution. |
| T-03-I | Information Disclosure | Unknown-placeholder behavior | mitigate | Unknown placeholders like `{Date}` (wrong case) or `{viewport_name}` (wrong key) are left literal in the output. This is the opposite of disclosure — it surfaces the typo on the filesystem (the user sees `screenshots/{Date}/...` and immediately recognizes the bug) instead of silently dropping the placeholder. |
</threat_model>
