<objective>
Build the prepare-pipeline orchestrator + barrel — `src/prepare/index.js`. This is the single import surface the CLI's `src/cli.js` consumes; the executor of 04-05 will not need to know which submodule (`animations.js` / `hide.js` / `scroll.js`) implements which PREP-*. The file re-exports `installAnimationGuards` from `./animations.js` (so the CLI's pre-navigation hook lives behind the same import path) and defines a new composing function `runPreparePipeline(page, prepareConfig)` that sequences the post-navigation steps in fixed order: hide → scrollPrime (gated) → extraDelay (unconditional, function-internal short-circuit). The orchestrator returns `{ hideSummary }` — informational pass-through for Phase 6 to surface later.

Purpose: The CLI in 04-05 needs ONE clean import statement covering both lifecycle halves of the prepare pipeline. Without an orchestrator, the CLI would import from 4 separate files (animations, hide, scroll for the two scroll-pair functions, and 4-of-5 functions total) and the lifecycle ordering would be encoded in the CLI rather than in a dedicated module. With this orchestrator, the CLI's diff is two import names and two call sites (one pre-nav, one post-nav). The composing function is INTENTIONALLY tiny — 04-PATTERNS §Risk 2 forbids a class hierarchy or hook-registration system; the four-line `if (prepareConfig.scrollPrime)` sequence from RESEARCH §Pattern 4 line 529-541 is the right shape.
