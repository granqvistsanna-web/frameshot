<objective>
Build the post-navigation scroll-prime + extra-delay module — two related exports in a single file. `scrollPrime(page)` scrolls the document from top to bottom in `window.innerHeight` steps with a 200ms wait between each step (so browser-native `loading="lazy"` IntersectionObservers can trigger their image fetches), then resets to `scrollTop = 0` so Phase 5's capture loop starts from a known position. `extraDelay(page, ms)` is a trivial `await page.waitForTimeout(ms)` wrapper that short-circuits on `ms <= 0` — it exists as a named export so the orchestrator (04-05) can wrap it in `console.time` for the SC #4 evidence check WITHOUT putting library code in the timing path.

Purpose: PREP-04 ("Scroll prime runs (scroll to bottom in viewport-height steps with wait, then scroll back to top) when enabled in config") and PREP-05 ("Configured `extraDelay` (ms) is honored before capture begins") are both Playwright timing primitives — neither involves DOM mutation beyond `window.scrollTo`. The key architectural choice is `behavior: 'instant'` over `behavior: 'smooth'`: 04-RESEARCH §Pitfall 6 and 04-PATTERNS §Risk 12 document that smooth scrolling animates over ~300ms and races with the 200ms inter-step wait — lazy-load IntersectionObservers fire at wrong positions, defeating the prime. Instant scrolling is deterministic. The two functions are co-located in `scroll.js` per 04-RESEARCH §Module layout recommendation: both are post-navigation timing helpers; both consume the Page handle; both are trivially small (~10 lines each). The dual-export pattern matches `src/config/schema.js` (which exports both `configSchema` and `formatZodError` from one file — 04-PATTERNS §scroll.js Two-export pattern).

Output: `src/prepare/scroll.js` — two named async exports, two `page.evaluate` calls inside the loop (one for geometry-read, one per iteration for the scroll), N+1 `page.waitForTimeout` calls (N inter-step waits + 0 or 1 extraDelay).

        ```
        const { innerHeight, totalHeight } = await page.evaluate(() => ({
          innerHeight: window.innerHeight,
          totalHeight: document.documentElement.scrollHeight,
        }));

        for (let y = 0; y < totalHeight; y += innerHeight) {
          await page.evaluate((targetY) => {
            window.scrollTo({ top: targetY, behavior: 'instant' });
          }, y);
          await page.waitForTimeout(INTER_STEP_WAIT_MS);
        }

        // Final reset to top — Phase 5's capture loop assumes scrollY=0 at start.
        await page.evaluate(() => {
          window.scrollTo({ top: 0, behavior: 'instant' });
        });
        ```

| `ms` (validated number from Phase 2) → `page.waitForTimeout(ms)` | Phase 2's zod schema enforces `z.number().int().min(0)` (schema.js:31). The `if (ms <= 0) return;` short-circuit defends against unexpected negatives. Playwright's waitForTimeout accepts any non-negative number. |
| `window.scrollTo` calls → page scroll position | Page-context callbacks scroll the document. Side-effect-only; no data flows back. The position reads (`window.scrollY` in the verify) read the same property. |
| `page.evaluate` geometry read → JS object back to Node | Returns `{ innerHeight, totalHeight }` as JSON-serialized object. Both are plain numbers; no exfiltration risk. |
| Playwright primitive errors → caller's catch | If the page closes mid-prime (e.g. browser crash), Playwright throws; the CLI's top-level handler catches and exits. No try/catch in this module. |
