<objective>
Build the post-navigation element-hiding module — a single async function that, given a Page and an array of CSS selectors, hides every matching element using `visibility: hidden !important` (preserving layout, critical for Phase 5's scroll-stitch math). The function returns a `{ matched, missed }` summary so future Phase 6 work can surface warnings about selectors that matched nothing, without coupling Phase 4 library code to terminal-UX concerns.

Purpose: PREP-03 ("Selectors in the config's `hide` list are hidden from the page before capture — sticky navs, banners, chat widgets") is the most mechanical of the five PREP requirements. Its sole architectural choice is the hide mechanism: `visibility: hidden` (preserves layout, removes visual presence) vs `display: none` (removes from layout, shrinks `document.scrollHeight`). 04-RESEARCH §Pitfall 5 and 04-PATTERNS §Risk 11 establish that Phase 5's scroll-stitch loop depends on consistent scroll height across the prepare→capture transition — `display: none` would shrink the document mid-pipeline and corrupt the stitch math. `visibility: hidden` is mandatory. The return shape `{ matched, missed }` exposes observability data without making library code own warning presentation (split per 04-RESEARCH §Architectural Responsibility Map: Phase 4 returns the data, Phase 6 prints it).

Output: `src/prepare/hide.js` — one named async export, one `page.evaluate` call with the selectors array passed as an argument (not inline-interpolated), no I/O beyond the Playwright Page argument.

      Step 1 — Single `page.evaluate` call passing selectors as the `arg` (RESEARCH §Pattern 2 lines 395-422 — verbatim usable):

        ```
        return page.evaluate((sels) => {
          let matched = 0;
          const missed = [];
          for (const sel of sels) {
            let nodes;
            try {
              nodes = document.querySelectorAll(sel);
            } catch {
              // querySelectorAll throws SyntaxError on invalid CSS syntax.
              // Lump syntax errors with zero-match into `missed` so a typo
              // in the config doesn't kill the whole prepare pipeline.
              missed.push(sel);
              continue;
            }
            if (nodes.length === 0) {
              missed.push(sel);
              continue;
            }
            for (const node of nodes) {
              // Inline-important pattern beats author stylesheets that may
              // declare higher specificity rules on the same element. A
              // simple `node.style.visibility = 'hidden'` would lose to a
              // page-level `.sticky-nav { visibility: visible }` declaration.
              node.style.setProperty('visibility', 'hidden', 'important');
              matched++;
            }
          }
          return { matched, missed };
        }, selectors);
        ```

        Notes:
        - The selectors array is passed as the SECOND argument to `page.evaluate`, NOT interpolated into the function source.         - The inner `try { document.querySelectorAll(sel) } catch { missed.push(sel); continue; }` catches `SyntaxError` (invalid CSS selector — e.g. `[unclosed[`).
