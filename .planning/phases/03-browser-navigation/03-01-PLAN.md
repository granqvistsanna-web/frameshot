<objective>
Build the headless Chromium launcher — a pure async function that, given a validated config, returns a `{ browser, context }` pair where the context is preconfigured with the right viewport pixels, deviceScaleFactor, and baseURL. This is the wave 1 half of Phase 3: zero CLI integration, zero navigation, just `chromium.launch()` + `browser.newContext()` + a typed error class. 03-02 (wave 2) layers navigation, the `--smoke` seam, and the `src/cli.js` wiring on top.

Purpose: CAP-01 (configured viewport) and CAP-02 (deviceScaleFactor) both map to a single Playwright API call — `browser.newContext({ viewport, deviceScaleFactor, baseURL })`. Isolating that in its own module keeps the surface tiny (~30 lines), makes the "context-before-browser close order" contract loud in JSDoc, and lets the navigator (03-02) consume `{ browser, context }` without inheriting any launch concerns. The BrowserError class mirrors Phase 2's ConfigError so the top-level catch in `src/cli.js` (wired by 03-02) has one consistent shape for all user-facing errors.

Output: `src/browser/launcher.js` — one named async function, one exported error class, no I/O beyond the Chromium subprocess.

## Module Root: `src/` (uniform across Phases 2 and 3)

The module root for framershot is `src/`. Phase 2's drafts already place modules under `src/` (`src/cli.js`, `src/config/load.js`, `src/config/schema.js`, `src/output/template.js`). Phase 3 honors that convention without exception — every new file in this phase lives under `src/`. There is no `lib/` directory in this project.

Rationale: (a) RESEARCH.md §Assumption A4 recommends this; (b) less CLI surface area for a personal v0.1 tool; (c) the `--smoke` modifier lives within the existing capture-flow code path so the lifecycle (launch → close in try/finally) is shared with the eventual non-smoke path that Phases 4+5 will build. The `--smoke` flag is implemented in 03-02 — recorded here so both planning artifacts share the same decision.
