      JSDoc block above the function MUST include:
      - One-line summary: "Open a new page in the context, navigate to `pageConfig.path` (joined against the context's baseURL), wait for network to settle (best-effort), and wait for fonts to be ready. Returns the Page."
      - `@param {import('playwright-chromium').BrowserContext} context` — the context returned by launchBrowser.

      Function body (RESEARCH.md §Pattern 2 verbatim — do not invent additional logic):

      Step 1 — Open a new page:
        ```
        const page = await context.newPage();
        ```

      Step 2 — Navigate with networkidle (timeout-tolerant):
        ```
        try {
          await page.goto(pageConfig.path, {
            waitUntil: 'networkidle',
            timeout: DEFAULT_NAV_TIMEOUT_MS,
          });
        } catch (err) {
          if (err.name === 'TimeoutError') {
            // The page DID start loading (goto returned a Response and the
            // timeout fired on the post-load networkidle wait). Continue —
            // fonts.ready below is the real gate, and Phase 4 prepare adds
            // further settling. We intentionally do NOT log here — Phase 6
            // owns terminal UX; if Phase 6 wants to warn on networkidle
            // timeout, it can wrap this call.
          } else {
            // SSL error, DNS failure, invalid URL, ERR_NAME_NOT_RESOLVED,
            // etc. — these are fatal.
            throw new BrowserError(
              `Failed to navigate to ${pageConfig.path} (resolved against context baseURL). ` +
              `Original error: ${err.message}`,
              { cause: err },
            );
          }
        }
        ```

        - `pageConfig.path` (NOT `config.baseUrl + pageConfig.path`) — the context's baseURL handles joining. RESEARCH.md §Don't Hand-Roll first row.
        - `waitUntil: 'networkidle'` — CAP-03 names this verbatim. RESEARCH.md §Summary explains that networkidle is officially DISCOURAGED upstream but CAP-03 is explicit, so we honor it literally with the timeout-tolerant guard.
        - `timeout: 15000` — RESEARCH.md §Pitfall 2.
        - `err.name === 'TimeoutError'` discriminator — Playwright's TimeoutError class has `.name === 'TimeoutError'`. The discriminator separates "DOM is up but network won't idle" (recoverable) from "URL never resolved" (fatal). RESEARCH.md §Pattern 2 second snippet.

      Step 3 — Wait for fonts (also timeout-tolerant; this is the CAP-04 implementation):
        ```
        try {
          await page.waitForFunction(() => document.fonts.ready, null, {
            timeout: DEFAULT_NAV_TIMEOUT_MS,
          });
        } catch (err) {
          if (err.name === 'TimeoutError') {
            // Same posture as networkidle: don't fail the run. A 15s font
            // wait covers all reasonable cases; if it times out, fall
            // through with whatever the page has rendered so far.
            // Playwright also internally awaits document.fonts.ready
            // before page.screenshot() (per issue #35200) — belt-and-braces.
          } else {
            throw new BrowserError(
              `Error waiting for fonts to be ready. Original error: ${err.message}`,
              { cause: err },
            );
          }
        }
        ```

        - `page.waitForFunction(...)` — NOT `page.evaluate(...)`. RESEARCH.md §Pitfall 6: waitForFunction is the documented-for-waiting form. Both technically work for `document.fonts.ready` (which is a Promise), but evaluate signals "extract a value" to the reader, while waitForFunction signals "poll for condition" and uses the navigation timeout semantics.
        - The second argument `null` is the optional `arg` parameter (no args needed for our predicate). Some examples omit it; including `null` matches the Playwright docs signature explicitly.
        - The third argument `{ timeout: ... }` overrides the default `actionTimeout` (which is the wrong default for this kind of wait — navigation timeout semantics are what we want).
        - Why timeout-tolerant: RESEARCH.md §Pitfall 4 — Framer uses `font-display: swap`, so the initial render shows fallback fonts and swaps to the web font when it loads. Most cases this is fast; if a font CDN is slow, falling through with fallbacks is a better UX than hanging indefinitely. Playwright also re-awaits `document.fonts.ready` internally before `page.screenshot()` (per issue #35200), so even on timeout the screenshot path has a final safety net.
        - **Note on CAP-04 coverage:** The hermetic data: URL fixture in Task 2 declares no `@font-face` rules, so its `document.fonts` collection is empty and `document.fonts.ready` resolves immediately. That means the hermetic verify proves only that this code path RUNS without error; it does not prove the code path BLOCKS for fonts. The blocking behavior is proven by the live-site manual gate (see `<manual_gates>` below).

      Step 4 — Return:

        ```
        return page;
        ```

        Note we do NOT close the page anywhere in this function — its lifecycle is bound to the context's, and the context is closed by the CLI layer (Task 3). Phase 4 and Phase 5 will consume the returned `page` directly.
