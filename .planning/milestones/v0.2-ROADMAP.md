# Milestone v0.2: Multi-viewport & Region Capture

**Status:** ✅ SHIPPED 2026-05-24
**Phases:** 7–8
**Total Plans:** 8

## Overview

v0.2 extends framershot beyond v0.1's single-page-single-viewport capture without compromising the prepare pipeline. The schema gains a `viewports: [...]` array (with backward-compatible singular-shape normalization) so one config + one `framershot capture` run produces N retina PNGs — each in its own Playwright browser context, fully isolated. On top of that, `regions: [...]` adds two new capture modes: by CSS selector (`element.screenshot()` with padding) and by from/to anchor pair (bounding box clip). A `--only=<region-name>` CLI flag with fail-fast validation isolates a single region per run. The hermetic smoke fixture pattern extends with two new YAMLs (`samples/smoke-multi.yaml`, `samples/smoke-regions.yaml`).

## Phases

### Phase 7: Multi-viewport Capture

**Goal**: User declares an array of viewports in a single config and one `framershot capture` run produces one full-page PNG per viewport, each with its own browser context.
**Depends on**: Phase 6 (v0.1 shipped)
**Requirements**: MULTI-01
**Plans**: 4 plans

Plans:

- [x] 07-01: Schema + loader — `viewportEntrySchema`/`viewportsSchema` + mutually-exclusive root refinement + singular→plural normalize transform (Wave 1)
- [x] 07-02: `runCapture` per-viewport loop + `launchBrowser(config, viewportEntry)` refactor + per-viewport-scoped `onProgress` events + array return shape (Wave 2)
- [x] 07-03: CLI ora adapter + server SSE adapter consume array return; per-viewport spinner prefix `[<viewport>]`; UI form stays single-viewport per D-05 (Wave 3)
- [x] 07-04: `samples/smoke-multi.yaml` hermetic 2-viewport fixture + human visual checkpoint (Wave 3, after 07-03)

**Success Criteria (5/5 PASS, verified 2026-05-24):**

1. ✓ A config with a `viewports: [...]` array of 2+ entries produces one full-page PNG per viewport in a single `framershot capture` run
2. ✓ The `{viewport}` placeholder in the output template resolves to the per-viewport `name` field so PNGs land at distinct paths (no overwrites)
3. ✓ Each viewport runs in its own Playwright browser context — no shared cookies, storage, or animation-shim state between viewports
4. ✓ A v0.1-shaped config with a single `viewport: { ... }` block still validates and runs unchanged (backward compatibility preserved)
5. ✓ The hermetic smoke fixture demonstrates a 2-viewport run producing 2 distinct PNGs end-to-end

**Details:**

The schema layer (`src/config/schema.js`) adds a root `.superRefine` that enforces mutual exclusivity between singular `viewport:` and plural `viewports:`, followed by a `.transform` that normalizes the singular shape into `viewports: [{...}]` with `name ?? 'default'`. Downstream code only reads `config.viewports[]` — `config.viewport` is gone post-transform. `runCapture` (`src/capture/runCapture.js`) wraps the v0.1 capture body in a `for (const vp of config.viewports)` loop with a per-iteration `try/finally` that closes the context then the browser before the next iteration; this is what guarantees per-viewport isolation. `launchBrowser` becomes 2-arg (`config, viewportEntry`), separating top-level baseURL+DSR from per-viewport geometry. Every `onProgress` event now carries `viewport: <name>` for scoped event consumption. The CLI ora adapter prefixes spinner text with `[<viewport>]` and emits one stdout line per result; the server SSE adapter returns `outputs: [{ outputPath, urlPath, viewportName }, ...]` (array shape).

### Phase 8: Region Capture

**Goal**: User declares named regions in config — by CSS selector or by from/to anchor pair — and captures each region as its own PNG, with `--only=<region-name>` to capture a single region instead of the full page.
**Depends on**: Phase 7
**Requirements**: REGION-01, REGION-02, REGION-03
**Plans**: 4 plans

Plans:

- [x] 08-01: Schema — `regionSchema` (selector-XOR-anchor `superRefine`) + root `regions:` field + duplicate-name + `{region}`-in-output cross-field refinements (Wave 1)
- [x] 08-02: New `src/capture/region.js` (`captureRegion` + `RegionError` + `padRect`/`unionRect`/`clampToDocument` helpers) + `{region}` placeholder in `resolveTemplate` (Wave 2)
- [x] 08-03: `runCapture` region loop + `resolveRegions` helper + `--only` fail-fast validation + Open-Q#1-A regions+full-page branch (Wave 3)
- [x] 08-04: `--only` commander option + `--smoke`/`--only` mutex + `formatError` `RegionError` Guard 4 + `serve-smoke.js` HTML extension + `samples/smoke-regions.yaml` fixture + human visual checkpoint (Wave 4)

**Success Criteria (5/5 PASS, verified 2026-05-24):**

1. ✓ A config with `regions: [{ name, selector, padding? }]` produces one PNG per selector-region — scrolled into view, prepare pipeline applied, padding honored
2. ✓ A config with `regions: [{ name, from, to, padding? }]` produces one PNG per anchor-region — clipped to the bounding box spanning the two anchor elements
3. ✓ Running `framershot capture <config> --only=<region-name>` captures only that single named region (skips full-page stitch and other regions); without the flag, full-page stitch behavior from v0.1 is unchanged
4. ✓ A missing selector / missing anchor / unknown `--only` name surfaces an actionable error via the existing `formatError` dispatcher (typed error, not a raw throw)
5. ✓ The hermetic smoke fixture demonstrates: (a) a selector-region capture, (b) an anchor-region capture, (c) the `--only` flag isolating one region

**Details:**

`regionSchema` uses a single `z.object` + `.superRefine` (not `z.union`) for selector-XOR-(from+to) — chosen to preserve actionable per-entry messages over union-flattening. Root-level refinements enforce duplicate-name detection and `{region}`-in-output cross-field validation (overwrite prevention), mirroring Phase 7's viewports shape. `captureRegion` (`src/capture/region.js`) is a single entry point covering both modes: selector branch does count-check → bbox pre-check → `scrollIntoViewIfNeeded` → `measureDocBox` → `padRect` → `clampToDocument` → screenshot; anchor branch resolves both anchors → `unionRect` → `padRect` → `clampToDocument` → screenshot. Module-private geometry helpers (`padRect`/`unionRect`/`clampToDocument`) are colocated and not exported. `RegionError` mirrors `BrowserError`'s typed-error shape. `resolveTemplate` (`src/output/template.js`) gains a `{region}` placeholder that slugifies the region name; omitted region (full-page entries) renders the placeholder as a literal (unknown-placeholder-stays-literal posture). `runCapture` nests a `for (const region of regions)` loop inside the per-viewport loop (cartesian product); `--only` validation happens upfront before any Chromium launch (fail-fast). The CLI gains `--only <region-name>` (commander required-value form) and a `--smoke`/`--only` mutex guard; `formatError` adds Guard 4 for `RegionError` mirroring Guard 2 (ConfigError) shape verbatim.

## Milestone Summary

**Decimal Phases:**

- None — v0.2 ran Phases 7 and 8 as planned, no insertions

**Key Decisions:**

- **D-03 (Phase 7-02): Loop in `runCapture`, callers stay dumb adapters** — CLI/server consume an array return shape; the per-viewport loop is internal to `runCapture` so adapters never iterate viewports themselves. Outcome: ✓ Good — CLI/server changes in Phase 7-03 were lean (just an array consumer + per-result formatting).
- **D-04 (Phase 7-02): `launchBrowser(config, viewportEntry)` two-arg signature** — Per-viewport geometry separated from top-level DSR + baseURL. Outcome: ✓ Good — kept the launcher pure and let `runCapture` own the per-iteration lifecycle.
- **D-05 (Phase 7-03): UI form stays single-viewport** — Multi-viewport ergonomics belong to the YAML config; the web UI form remains single-viewport for daily-use simplicity. Outcome: ✓ Good — kept the UI scope tight.
- **D-02 precedent applied to regions (Phase 8-01): `{region}`-in-output cross-field refinement** — When `regions:` is present, the output template must include `{region}` (or `{date}` with timestamps) to prevent overwrites. Mirrors Phase 7's `{viewport}` precedent. Outcome: ✓ Good — caught misconfigurations at validation time.
- **Open-Q#1 lock A (Phase 8-03): regions + no `--only` = N regions + 1 full-page per viewport** — When `regions:` is present without `--only`, capture produces both each region AND the full-page stitch. Locked over the "regions replace full-page" alternative. Outcome: ✓ Good — preserves v0.1 full-page behavior as the default while adding regions as an additive surface.
- **Single `z.object` + `.superRefine` over `z.union` for regions (Phase 8-01)** — Preserves actionable per-entry error messages; `z.union` would flatten errors and lose the per-region name surface. Outcome: ✓ Good — Phase 8 verification confirmed all 4 error paths (missing selector / missing anchor / unknown `--only` / etc.) produce clean `Error:` prefix output via Guard 4.

**Issues Resolved:**

- Per-viewport prepare-pipeline isolation verified — `.does-not-exist` warning fires twice in the live multi-viewport run (once per viewport), observable proof of per-iteration `installAnimationGuards`
- v0.1 backward compatibility preserved end-to-end: `samples/smoke.yaml --smoke` still produces a single PNG with singular `viewport:` shape
- `--only` fail-fast: invalid region names throw `RegionError` BEFORE Chromium launch (no wasted browser cold-start)

**Issues Deferred:**

- Multi-page per config (MULTI-02) — out of v0.2 scope; will pick up when daily use surfaces the need
- `--viewport=<name>` and `--pages=<list>` CLI filters (MULTI-03, MULTI-04) — YAGNI until multi-viewport surface is used in anger
- Diff mode (DIFF-01), pre-capture hooks (HOOK-01), auth (AUTH-01) — all carried forward from v0.1 deferred list

**Technical Debt Incurred:**

- None observed. Phase 7 and 8 stayed within the existing module layout (`src/config/`, `src/capture/`, `src/cli/`, `src/output/`); new modules (`src/capture/region.js`) follow the established colocated-helper pattern; the typed-error dispatcher absorbed `RegionError` as Guard 4 without restructuring.
